## 解决的问题

传统预约流程通常依赖人工沟通，容易出现信息反复确认、价格口径不一致、可预约时段无法实时校验等问题。HairCutAgent 将自然语言对话接入后端预约系统，让用户可以用聊天方式完成预约、改期、取消、价格查询和发型咨询。

项目重点体现三类能力：

- **对话到业务动作**：将用户的自然语言输入拆解为意图、参数和具体操作，而不是只生成一段回答。
- **状态一致性**：预约记录、服务价格和营业规则由后端统一维护，AI 回复与后台数据保持一致。
- **可控的异常处理**：对缺失字段、预约冲突、非营业时间、节假日和记录不存在等情况提供明确分支，降低纯 Prompt 处理业务流程的不确定性。

## 工作流设计

Dify 侧将用户输入分为多个业务意图：

| 意图 | 场景 |
| --- | --- |
| GREETING | 打招呼、开场 |
| BOOKING | 预约、改期、取消、查预约、查时段 |
| HAIR_RECOMMEND | 发型推荐、脸型/发质建议 |
| PRICE_QUERY | 价格、收费标准、服务项目 |
| STORE_QUERY | 门店位置、有哪些店 |
| END / ABUSE / OTHER | 告别、攻击性输入、兜底 |

预约链路采用“意图识别 → 参数提取 → 必填字段校验 → HTTP 调用 → 结果回复”的结构。  
发型推荐链路采用 6 个维度收集：性别、脸型、发质、长度、风格偏好、打理成本，再进入知识库检索与推荐生成。

## 后端设计

FastAPI 后端负责把 Dify 的自然语言结果落成确定性的业务操作，并把结果暴露给前端页面：

- `/dispatch` 作为 Dify HTTP 节点统一入口，根据 `action` 分发业务。
- `bookings` 保存预约记录，支持按手机号或预约单号查询。
- `slots` 管理设计师可预约时段，启动时自动补齐未来时段。
- `services` 保存服务项目与价格，供 AI 价格查询和后台管理共用。
- `business_hours` 保存门店营业规则，用于拦截休息日、午休和节假日预约。
- `static/admin.html` 提供轻量后台，前台对话和后台数据保持联动。
- `static/experience.html` 作为用户体验层，展示设计师状态、预约结果查询和移动端适配后的对话入口。

## 设计取舍

这个项目把 AI 能力、业务状态和前端体验做了明确分工：

- Dify 负责意图识别、多轮信息补齐、知识检索和自然语言回复。
- FastAPI 负责预约校验、营业规则、时段冲突检测和数据读写。
- SQLite 作为统一数据源，保证价格、预约记录和后台管理数据一致。
- HTML / CSS / JS 负责项目展示、用户体验页和管理后台交互，承接真实演示入口。

这样的拆分主要解决三个问题：

- LLM 负责什么，后端负责什么。
- 用户体验页、前台对话、后台管理和数据库状态如何联动。
- AI 接入业务系统时，规则控制和异常处理如何落地。

最近一轮工程化补强还补上了两个面向真实部署的细节：

- Admin 管理接口增加了基于环境变量的 Bearer Token 鉴权，未配置时默认禁用管理接口，避免后台接口裸露。
- 数据库连接统一收口到上下文管理器，并补充了 pytest + TestClient 回归测试，覆盖公共接口、管理接口鉴权和核心业务分支。

本轮演示补强还增加了三个更适合面试展示的结果：

- 新增独立用户体验页，主入口不再直接暴露原始 Dify 对话，而是先展示可讲解的前端交互层。
- 用户体验页支持手机端排版调整，保证在移动设备上优先完成“开始对话 / 查询预约”的任务路径。
- 管理后台切换设计师可预约状态后，用户体验页会同步展示最新状态，证明前后端读取的是同一套业务数据。

## 只读 MCP 适配层

`mcp_server.py` 使用官方 Python MCP SDK，为外部 AI 客户端提供 3 个只读业务查询工具，并复用 FastAPI 服务使用的同一份 SQLite 数据：

- `list_services`：查询启用中的服务、价格和时长。
- `get_business_hours`：查询营业时间、午休、每周休息日和节假日规则。
- `get_available_slots`：查询实时可预约时段，并明确返回闭店状态。

适配层不开放预约创建、改期、取消或后台管理等写操作，所有状态变更仍由现有 FastAPI 业务层控制。默认使用 stdio；服务器侧可使用仅绑定 `127.0.0.1:8004` 的 Streamable HTTP，不直接暴露公网。

本地 MCP 客户端可直接使用默认 stdio：

```bash
python mcp_server.py
```

服务器侧私有 Streamable HTTP：

```bash
MCP_TRANSPORT=streamable-http ./start_mcp_server.sh
```

## 实现边界

- 预约主链路由 Dify Workflow 在意图识别和字段校验后固定调用 HTTP API，不描述为模型自主 Function Calling。
- MCP 是后续增加的独立只读适配层，用于标准化读取实时业务状态，不替代原有预约写链路。
- 本项目定位为可运行、可验证的 AI 业务系统原型，不虚构生产事故、生产流量或未实现的多 Agent 能力。
