# HairCutAgent - 理发店智能预约助手

HairCutAgent 是一个面向门店预约场景的 AI 业务系统展示项目，重点展示 Dify 工作流如何和 FastAPI、SQLite 及轻量管理后台组合，完成预约、改期、取消、价格查询和发型推荐等真实业务链路。

## 项目地址

- 展示页：`http://101.43.56.2:8899`
- 在线体验：通过展示页中的“立即体验”按钮跳转

## 解决的问题

传统预约流程通常依赖人工沟通，容易出现信息反复确认、价格口径不一致、可预约时段无法实时校验等问题。HairCutAgent 将自然语言对话接入后端预约系统，让用户可以用聊天方式完成预约、改期、取消、价格查询和发型咨询。

项目重点体现三类能力：

- **对话到业务动作**：将用户的自然语言输入拆解为意图、参数和具体操作，而不是只生成一段回答。
- **状态一致性**：预约记录、服务价格和营业规则由后端统一维护，AI 回复与后台数据保持一致。
- **可控的异常处理**：对缺失字段、预约冲突、非营业时间、节假日和记录不存在等情况提供明确分支，降低纯 Prompt 处理业务流程的不确定性。

## 核心功能

| 模块 | 能力 |
| --- | --- |
| 智能预约 | 收集姓名、手机号、门店、服务、日期和时间，缺什么追问什么 |
| 预约管理 | 支持创建预约、改期、取消、查询已有预约 |
| 价格查询 | 从后端服务项目表实时读取价格，避免静态话术过期 |
| 门店规则 | 支持营业时间、午休、休息日、节假日和多门店配置 |
| 发型推荐 | 通过 Dify 知识库检索与重排，根据用户条件生成推荐 |
| 管理后台 | 管理设计师、服务项目、营业时间和预约记录 |

## 工作流设计

Dify 侧将用户输入分为多个业务意图：

| 意图 | 场景 |
| --- | --- |
| GREETING | 打招呼、开场 |
| BOOKING | 预约、改期、取消、查预约、查时段 |
| HAIR_RECOMMEND | 发型推荐、脸型/发质建议 |
| PRICE_QUERY | 价格、收费标准、服务项目 |
| STORE_QUERY | 门店位置、有哪些店 |
| END / ABUSE / OTHER | 告别、攻击性输入、兜底 |

预约链路采用“意图识别 → 参数提取 → 必填字段校验 → HTTP 调用 → 结果回复”的结构。  
发型推荐链路采用 6 个维度收集：性别、脸型、发质、长度、风格偏好、打理成本，再进入知识库检索与推荐生成。

## 后端设计

FastAPI 后端负责把 Dify 的自然语言结果落成确定性的业务操作：

- `/dispatch` 作为 Dify HTTP 节点统一入口，根据 `action` 分发业务。
- `bookings` 保存预约记录，支持按手机号或预约单号查询。
- `slots` 管理设计师可预约时段，启动时自动补齐未来时段。
- `services` 保存服务项目与价格，供 AI 价格查询和后台管理共用。
- `business_hours` 保存门店营业规则，用于拦截休息日、午休和节假日预约。
- `static/admin.html` 提供轻量后台，前台对话和后台数据保持联动。

## 设计取舍

这个项目把 AI 能力和业务状态做了明确分工：

- Dify 负责意图识别、多轮信息补齐、知识检索和自然语言回复。
- FastAPI 负责预约校验、营业规则、时段冲突检测和数据读写。
- SQLite 作为统一数据源，保证价格、预约记录和后台管理数据一致。

这样的拆分主要解决三个问题：

- LLM 负责什么，后端负责什么。
- 前台对话、后台管理和数据库状态如何联动。
- AI 接入业务系统时，规则控制和异常处理如何落地。

最近一轮工程化补强还补上了两个面向真实部署的细节：

- Admin 管理接口增加了基于环境变量的 Bearer Token 鉴权，未配置时默认禁用管理接口，避免后台接口裸露。
- 数据库连接统一收口到上下文管理器，并补充了 pytest + TestClient 回归测试，覆盖公共接口、管理接口鉴权和核心业务分支。
