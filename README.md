# HairCutAgent - 理发店智能预约助手

HairCutAgent 是一个面向理发店预约场景的 AI Agent 作品集项目。它用 Dify 工作流理解用户的自然语言诉求，用 FastAPI + SQLite 承接真实预约数据，并提供一个轻量管理后台，让前台 AI 助手和后台业务数据保持同步。

这个项目不是一个只会回答问题的聊天 Demo，而是一个可演示的门店预约业务系统原型。

## 解决的问题

传统预约流程通常依赖人工沟通，容易出现信息反复确认、价格口径不一致、可预约时段无法实时校验等问题。HairCutAgent 将自然语言对话接入后端预约系统，让用户可以用聊天方式完成预约、改期、取消、价格查询和发型咨询。

项目重点验证三类能力：

- **对话到业务动作**：将用户的自然语言输入拆解为意图、参数和具体操作，而不是只生成一段回答。
- **状态一致性**：预约记录、服务价格和营业规则由后端统一维护，AI 回复与后台数据保持一致。
- **可控的异常处理**：对缺失字段、预约冲突、非营业时间、节假日和记录不存在等情况提供明确分支，降低纯 Prompt 处理业务流程的不确定性。

## 核心功能

| 模块 | 能力 |
| --- | --- |
| 智能预约 | 收集姓名、手机号、门店、服务、日期和时间，缺什么追问什么 |
| 预约管理 | 支持创建预约、改期、取消、查询已有预约 |
| 价格查询 | 从后端服务项目表实时读取价格，避免静态话术过期 |
| 门店规则 | 支持营业时间、午休、休息日、节假日和多门店配置 |
| 发型推荐 | 通过 Dify 知识库检索与重排，根据用户条件生成推荐 |
| 管理后台 | 管理设计师、服务项目、营业时间和预约记录 |

## 技术架构

```text
用户对话
  ↓
Dify Advanced Chat Workflow
  - 意图分类
  - 多轮参数提取
  - 字段完整性校验
  - RAG 发型推荐
  - 自然语言回复生成
  ↓
FastAPI 后端
  - /dispatch 统一业务入口
  - 预约、改期、取消、查询
  - 服务价格与管理后台接口
  ↓
SQLite 数据库
  - designers
  - services
  - business_hours
  - slots
  - bookings
```

## 工作流设计

Dify 侧将用户输入分为多个业务意图：

| 意图 | 场景 |
| --- | --- |
| GREETING | 打招呼、开场 |
| BOOKING | 预约、改期、取消、查预约、查时段 |
| HAIR_RECOMMEND | 发型推荐、脸型/发质建议 |
| PRICE_QUERY | 价格、收费标准、服务项目 |
| STORE_QUERY | 门店位置、有哪些店 |
| END / ABUSE / OTHER | 告别、攻击性输入、兜底 |

预约链路采用“意图识别 → 参数提取 → 必填字段校验 → HTTP 调用 → 结果回复”的结构。  
发型推荐链路采用 6 个维度收集：性别、脸型、发质、长度、风格偏好、打理成本，再进入知识库检索与推荐生成。

## 后端设计

FastAPI 后端负责把 Dify 的自然语言结果落成确定性的业务操作：

- `/dispatch` 作为 Dify HTTP 节点统一入口，根据 `action` 分发业务。
- `bookings` 保存预约记录，支持按手机号或预约单号查询。
- `slots` 管理设计师可预约时段，启动时自动补齐未来时段。
- `services` 保存服务项目与价格，供 AI 价格查询和后台管理共用。
- `business_hours` 保存门店营业规则，用于拦截休息日、午休和节假日预约。
- `static/admin.html` 提供轻量后台，便于面试演示“前台 AI + 后台管理”的联动效果。

## 项目说明

HairCutAgent 的设计重点是把 AI 对话接入一个可验证的业务流程，而不是停留在 FAQ 或闲聊层面。Dify 负责处理自然语言理解、多轮信息补齐和回复生成；FastAPI 负责确定性的业务校验、状态变更和数据查询；SQLite 作为统一数据源保存预约、服务价格、营业规则和可预约时段。

这种拆分让系统具备两个特点：

- **可控性**：预约、改期、取消等关键操作由后端规则和数据库状态决定，降低纯 Prompt 处理业务逻辑的不确定性。
- **可演示性**：前台对话创建或修改预约后，后台管理页面可以直接看到数据变化；后台调整服务价格后，AI 价格查询也会读取最新数据。

项目展示时可以重点关注：

1. Dify 工作流如何拆分意图分类、参数提取、字段校验和异常分支。
2. AI 与后端如何分工，让 LLM 处理语言问题，让后端处理业务状态。
3. 预约、改期、取消、查预约、价格查询和后台改价联动这些结果如何被现场验证。

## 原型说明

这是一个用于作品集展示的业务系统原型，重点验证 AI Agent 与后端业务系统的集成方式。仓库中未包含生产环境需要的用户认证、权限管理、审计日志、支付集成和隐私合规流程。
