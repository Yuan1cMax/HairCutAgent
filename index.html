What it solves

从"聊天"推进到"可执行预约"

项目重点不是生成一段客服话术，而是把自然语言输入转成确定性的业务动作，并让结果进入同一套预约数据，再同步反馈到用户页和管理后台。

多轮预约补齐

识别用户意图后提取姓名、手机号、门店、服务、日期和时间，缺失字段按优先级追问。

预约生命周期

支持创建预约、改期、取消、查询已有预约，并对不存在记录返回明确业务提示。

实时价格查询

服务项目和价格来自后端数据表，后台调整后，AI 查询结果同步变化。

营业规则校验

支持门店营业时间、午休、休息日和节假日校验，避免无效预约进入数据库。

发型推荐

按性别、脸型、发质、长度、风格和打理成本逐步补齐条件，再进入知识库检索与推荐生成，适合继续打磨成更完整的咨询链路。

前端交互页

独立用户体验页承接 Dify 对话入口、设计师状态展示和预约结果查询，并兼顾桌面端与手机端浏览。

后台联动

提供设计师、服务项目、营业时间和预约记录管理；设计师可预约状态调整后，前端展示页可实时同步。

只读 MCP 适配

通过 MCP 暴露服务项目、门店营业规则和可预约时段，让外部 AI 客户端读取同一份实时业务状态；不开放写操作。

Workflow

工作流拆分让业务更可控

预约场景有必填字段、异常分支和状态变更，使用 Dify 工作流拆成多个节点比单一 Prompt 更稳定、更容易调试。

1

意图识别

区分预约、价格查询、门店查询、发型推荐、告别和兜底输入。

2

参数提取

从用户输入和多轮上下文中提取日期、时间、门店、服务和联系方式。

3

字段校验

检查必填项、手机号格式、营业时间和节假日，必要时只追问一个字段。

4

业务执行

通过 FastAPI 写入或查询 SQLite，再由回复节点生成自然语言结果。

Architecture

AI 负责理解，后端负责状态，前端负责体验

Dify 处理自然语言理解、Slot 补齐和回复生成；FastAPI 处理确定性的数据库查询、预约冲突检测和营业规则校验；前端体验页负责把状态变化展示给真实用户。

SQLite保存 designers、services、business_hours、slots、bookings 等核心业务表。
MCP以只读工具方式复用同一 SQLite 数据源，提供服务、营业规则和可预约时段查询。
Experience UI独立用户体验页内嵌 Dify 对话，同时展示设计师状态、预约查询入口和移动端适配结果。
Admin UI轻量管理后台，负责查看预约记录、维护服务价格和营业时间。

项目把工作流编排、后端接口、数据库状态、用户前端和管理后台串起来，形成一个可验证、可演示、可公网访问的业务闭环。

HTML / CSS / JS实现项目展示页、用户体验页和轻量后台，并补齐预约结果查询、状态同步与移动端适配。
Nginx / Ubuntu部署 Dify、FastAPI 后端和静态页面，形成公网可访问链路。
GitHub公开仓库完成历史清理、运行产物忽略和敏感信息检查。
Prompt Engineering将预约口语表达转成可执行字段，并控制追问顺序和兜底话术。
Source Code

后端用确定性逻辑承接 AI 输出

LLM 负责理解用户怎么说，后端负责判断能不能做、怎么落库、如何返回业务状态。下面是项目中的核心分发思路节选。

main.pyFastAPI / dispatch
@app.post("/dispatch")
def dispatch(req: dict):
    action = req.get("action")
    phone = req.get("phone")
    date = req.get("date")

    # 先做营业时间、节假日、午休等确定性校验
    if action in ("create", "reschedule") and date:
        if is_holiday(date, req.get("store")):
            return {"status": "holiday"}

    # 再按业务动作分发，避免把状态变更交给 Prompt
    if action == "query_booking":
        return query_booking(phone=phone)
    if action == "create":
        return create_booking(...)
    if action == "cancel":
        return cancel_by_phone(phone)

预约落库

对话创建真实记录

用户补齐姓名、手机号、门店、服务和时间后，后端创建预约记录，管理后台可直接查看。

价格同步

后台数据影响 AI 回复

服务价格从后端接口读取，后台调整服务项目后，价格查询链路会返回最新结果。

前端联动

管理后台改动可同步到用户页

在管理后台切换设计师可预约状态后，用户体验页的设计师列表会刷新展示最新状态，证明前后端数据共用同一套来源。

异常兜底

业务失败也有明确状态

预约冲突、非营业时间、节假日或记录不存在时，系统返回可解释的业务提示，而不是泛化回答。

工程化补强

后台接口与管理鉴权补齐

管理接口增加 Bearer Token 鉴权，Token 从服务器私有环境加载，避免管理接口裸露或把敏感配置带入仓库。
